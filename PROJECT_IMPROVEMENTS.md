# Project improvement backlog

Ниже — практичный список улучшений, которые стоит внедрять поэтапно.

## 1) Надёжность и эксплуатация

- [x] **Graceful shutdown для сервера метрик**  
  Сейчас метрики поднимаются через `http.ListenAndServe` в горутине без контролируемого `Shutdown`.  
  Рекомендуется перейти на `http.Server` + `Shutdown(ctx)` при остановке процесса, чтобы избежать «висящих» соединений и шумных логов при завершении.

- [x] **Таймауты записи для SMTP-соединений**  
  В `internal/smtp/server.go` выставляется только read deadline. Добавить write deadline перед ответами клиенту, чтобы защищаться от медленных/зависших получателей.

- [x] **Ограничение параллелизма входящих SMTP-сессий**  
  Сейчас на каждое соединение создаётся отдельная горутина без верхнего лимита. Добавить semaphore/worker-pool, чтобы предотвратить деградацию под всплеском входящего трафика.

## 2) Наблюдаемость (observability)

- [x] **Request/trace correlation в логах**  
  Добавить request/session ID для SMTP-сессий и прокидывать его в логи relay/send. Это ускорит расследование инцидентов.

- [ ] **Расширить метрики latency-гистограммами**  
  Помимо счётчиков, добавить гистограммы:
  - парсинга email;
  - отправки в MAX API;
  - end-to-end relay времени.

- [ ] **Health/readiness endpoints**  
  На metrics HTTP-сервер добавить `/healthz` и `/readyz` (например, доступность alias-хранилища и проверка токена/доступности MAX API).

## 3) Конфигурация и безопасность

- [ ] **Валидация конфигурации "fail fast"**  
  Усилить `config.Load()` проверками согласованности параметров (например, `MAX_SENDER_MODE=botapi` требует `MAX_BOT_TOKEN`; разумные пределы для retry и message size).

- [ ] **Опциональный SMTP AUTH strict mode**  
  Сейчас `AUTH PLAIN` принимает любые креды (ожидаемо для relay-mode). Полезно добавить опцию строгой авторизации для деплоев в менее доверенной сети.

- [ ] **Ротация/маскирование чувствительных данных в логах**  
  Зафиксировать правило: никогда не логировать токены, сырой SMTP DATA целиком и персональные данные без необходимости.

## 4) Качество кода и тестирование

- [ ] **Race detector в CI (`go test -race ./...`)**  
  Особенно важно для SMTP-сервера, бота и метрик, где есть конкурентный доступ.

- [ ] **Негативные и property-based тесты парсеров**  
  Для recipient/email parser добавить генеративные сценарии (fuzz/property testing) на malformed input.

- [ ] **Контрактные интеграционные тесты MAX API клиента**  
  Для режима `botapi` добавить тест-контракты с mock HTTP server: ретраи, коды ошибок, rate-limit ответы.

## 5) Производительность

- [ ] **Профилирование горячих путей**  
  Под synthetic нагрузкой измерить CPU/allocs (pprof) и оптимизировать:
  - парсинг DATA;
  - сериализацию payload для MAX;
  - повторные аллокации в SMTP цикле.

- [ ] **Batch/queue режим отправки (опционально)**  
  Если ожидается высокий поток, рассмотреть внутреннюю очередь с bounded capacity + backpressure для контролируемой деградации.

## 6) Документация и DX

- [ ] **Runbook для инцидентов**  
  Кратко описать: что проверять при росте `relay failure`, как диагностировать проблемы MAX API, какие метрики/логи смотреть в первую очередь.

- [ ] **Примеры production-конфигураций**  
  Добавить `.env.example` для dev/stage/prod профилей и рекомендации по портам, retry, лимитам сообщений.

- [ ] **Architecture diagram**  
  Небольшая схема потока: SMTP ingress → parse → recipient resolve → MAX send → metrics/logging.
